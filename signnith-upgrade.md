# Signnith 서비스 고도화 전략: 100인 동시 접속 최적화 (무료 티어 기준)

본 문서는 Signnith 주식 분석 서비스의 동시 접속 용량을 무료 인프라 내에서 극대화하기 위한 단계별 고도화 전략을 담고 있습니다.

## 1. 아키텍처 개요: 3단계 다층 캐싱 (Multi-layer Caching)

무료 서버(Render)의 자원 한계를 극복하기 위해 데이터를 서버가 아닌 **DB와 클라이언트(브라우저)로 최대한 분산**시킵니다.

### 계층 1: 브라우저 분산 캐시 (IndexedDB)
*   **전략**: 사용자가 조회한 모든 분석 리포트를 브라우저의 전용 DB(IndexedDB)에 영구 저장합니다.
*   **효과**: 동일 사용자가 재방문 시 서버 호출을 0회로 만듭니다. 서버 CPU 및 아웃바운드 트래픽 비용을 획기적으로 절감합니다.

### 계층 2: 공유 영구 캐시 (Supabase L2 Store)
*   **전략**: 서버 인스턴스가 달라지거나 재시작되어도 유지되도록 분석 결과를 Supabase 테이블에 캐싱합니다.
*   **효과**: 인스턴스 간 연산 결과를 공유하여 외부 API(DART 등) 호출 횟수를 최소화합니다.

### 계층 3: 지능형 메모리 캐시 (L1 LRU)
*   **전략**: 서버 RAM 내에 가장 자주 쓰이는 데이터만 유지하는 LRU(Least Recently Used) 캐시를 적용합니다.
*   **효과**: 빈번한 요청에 대해 수 밀리초(ms) 단위의 초고속 응답을 제공하며, 메모리 부족(OOM) 현상을 방지합니다.

---

## 2. 데이터 성격별 캐시 정책 (Tiered TTL)

데이터의 '신선도' 요구사항에 맞춰 저장 위치와 만료 시간을 차등 적용합니다.

| 데이터 유형 | 저장 위치 | 만료 시간(TTL) | 비고 |
| :--- | :--- | :--- | :--- |
| **현재가/거래량** | L1 (RAM) | 2분 | 장 중 실시간성 확보 |
| **차트/캔들 데이터** | L2 (Supabase) | 4시간 | 기술적 분석용 데이터 리프레시 |
| **심층분석/AI 리포트** | L2/L3 (IndexedDB) | 24~48시간 | 연산량이 크므로 최대한 길게 유지 |
| **종목 리스트/마스터** | L3 (IndexedDB) | 7일 | 거의 변하지 않는 정적 데이터 |

---

## 3. 부하 분산 전략 (Zero-Cost Optimization)

### 🛡️ Cache Stampede 방지 (Request Coalescing)
*   동시에 여러 명이 동일 종목을 요청할 경우, 서버는 외부 호출을 **단 1회**만 수행하고 결과를 모든 대기자에게 동시에 반환(Mutex Lock)하여 API 쿼터를 보호합니다.

### 🛡️ 스크래핑 IP 차단 우회 (Smart Jitter)
*   유료 프록시 없이 Yahoo/Naver 차단을 피하기 위해 요청 간에 랜덤한 지연(Jitter)을 삽입하고, 동시 요청 수를 제한하는 지능형 재시도 로직을 적용합니다.

### 🛡️ SWR (Stale-While-Revalidate) 아키텍처
*   서버가 잠들어 있거나 데이터가 오래되었을 때, **일단 과거 데이터를 즉시 보여준 뒤** 백그라운드에서 데이터를 갱신합니다. 사용자는 '무한 대기' 없이 즉각적인 UI 경험을 얻습니다.

## 4. 인프라 운영 최적화 (Supabase Free)

*   **자동 퍼지(Auto-Purge)**: 무료 DB 용량(500MB) 유지를 위해 48시간 이상 지난 캐시 데이터는 DB 트리거를 통해 자동으로 삭제됩니다.
*   **데이터 압축**: JSON 데이터를 압축하여 저장함으로써 용량 효율을 300% 이상 향상시킵니다.
